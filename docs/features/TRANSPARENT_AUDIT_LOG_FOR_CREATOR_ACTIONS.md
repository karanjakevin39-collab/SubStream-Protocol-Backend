# Transparent Audit Log For Creator Actions

## Overview

This feature adds a durable, append-only creator audit trail for the business-changing actions that matter for legal, governance, reporting, and tax workflows.

The backend now records immutable audit entries whenever a creator:

- updates FlowRate
- changes a video's visibility
- modifies a Co-op Split

Creators can also retrieve and export their own audit history as CSV or PDF.

## Why This Exists

Creators need a trustworthy historical record of key account and revenue decisions. This helps with:

- compliance evidence
- internal governance review
- dispute resolution
- tax and reporting workflows

## Audited Actions

- `FLOW_RATE_UPDATED`
- `VIDEO_VISIBILITY_CHANGED`
- `COOP_SPLIT_MODIFIED`

## Stored Data

Each audit entry stores:

- `id`
- `creator_id`
- `action_type`
- `entity_type`
- `entity_id`
- `timestamp`
- `ip_address`
- `metadata`
- `created_at`

Metadata is intentionally limited to safe change context such as before/after values and split summaries. Secrets, access tokens, and internal credentials are not stored.

## Immutability Model

The audit log is append-only in application behavior.

- audit records are only created with `INSERT`
- there are no update routes for audit records
- there are no delete routes for audit records
- timestamps are generated by the server
- creator identity comes from the authenticated token, not client input

## Persistence

The backend uses a dedicated SQLite table named `creator_audit_logs`.

Related creator mutation tables are:

- `creator_settings`
- `videos`
- `coop_splits`

Creator mutations and audit writes are executed inside the same SQLite transaction. If audit persistence fails, the parent mutation is rolled back so compliance-sensitive changes are not silently applied without a trail.

## Endpoints

### Update FlowRate

- `PATCH /api/creator/flow-rate`

Request body:

```json
{
  "flowRate": "250",
  "currency": "USDC"
}
```

### Update Video Visibility

- `PATCH /api/creator/videos/:videoId/visibility`

Request body:

```json
{
  "visibility": "public"
}
```

Supported visibility values:

- `public`
- `private`
- `subscribers_only`
- `unlisted`

### Update Co-op Split

- `PATCH /api/creator/coop-splits/:splitId`

Request body:

```json
{
  "splits": [
    {
      "walletAddress": "G...",
      "percentage": 60
    }
  ]
}
```

### List Audit Log

- `GET /api/creator/audit-log`

### Export Audit Log

- `GET /api/creator/audit-log/export?format=csv`
- `GET /api/creator/audit-log/export?format=pdf`

Only the authenticated creator can retrieve or export their own audit data.

## Authorization Model

Creator endpoints require a signed bearer token with:

- issuer matching the configured backend issuer
- audience matching the configured creator audience
- role `creator`
- subject set to the creator identifier

The creator id used for audit entries is always derived from the verified token.

## IP Capture Notes

IP addresses are derived from `req.ip` and then normalized. This respects Express proxy handling only when trusted proxy mode is explicitly enabled by the app. If no valid IP can be determined, the stored fallback is `unknown`.

## Export Behavior

### CSV

The CSV export:

- includes a header row
- includes normalized audit columns
- quotes all cells
- prefixes dangerous spreadsheet formula prefixes with `'` to reduce CSV injection risk

### PDF

The PDF export:

- is intentionally simple and compliance-oriented
- includes creator id and export timestamp
- renders each entry in a readable text-based structure
- avoids embedding active content

## Operational Notes

- Set `DATABASE_FILENAME` to choose the SQLite database location.
- Set `CREATOR_AUTH_SECRET` to a strong random secret in non-development environments.
- Set `CDN_TOKEN_SECRET` and Soroban configuration independently for the secure streaming flow.

## Limitations And Future Extensions

- The current repository did not contain an existing auth or ORM layer, so the implementation uses a focused SQLite persistence layer and signed creator tokens.
- Filtered audit log exports can be added later if needed.
- If the project adopts a larger database stack later, the append-only audit schema and service boundaries are ready to migrate.
